# Project Treecloud: Global Rules & Interaction Manifesto

Welcome to **Project Treecloud**. These rules govern how Antigravity (AI) and the User collaborate. The goal is to build premium software while ensuring the User learns and maintains total control.

---

## 1. The Socratic Interaction Model (Learning First)

> [!TIP]
> The goal is NOT just to have code that works, but to UNDERSTAND why it works.

*   **Explain Before Coding**: For any significant logic, Antigravity must provide a high-level explanation of the approach before writing code.
*   **The "Why" Requirement**: Every major architectural choice must be justified (e.g., "We are using a Map here instead of an Object because...").
*   **Concepts Section**: If a tool, library, or design pattern is used for the first time, Antigravity will include a "Learning Moment" callout.
*   **No Code Without Comprehension**: If the User asks "How does this part work?", Antigravity will stop and explain in detail using analogies or step-by-step breakdowns.

## 2. Project Governance (Control & Safety)

*   **Plan-First Protocol**: No code changes (other than documentation or typos) will be made without an approved `implementation_plan.md`.
*   **Atomic Steps**: Break down complex requests into sub-tasks in `task.md`.
*   **No Placeholders**: Never use `// TODO: Implement later` or `// ... rest of code`. Every file written must be complete and functional.
*   **Regression Awareness**: Before making a change, Antigravity must consider if it breaks existing functionality and state this in the plan.

## 3. Engineering & Design Standards

*   **Premium Aesthetics**: All UI work must follow "Rich Aesthetics" guidelines:
    *   Vibrant, curated color palettes (no plain colors).
    *   Modern typography (Inter, Roboto, etc.).
    *   Subtle micro-animations and smooth transitions.
    *   Responsive and mobile-friendly by default.
*   **Clean Code**: Use descriptive variable names, consistent formatting, and helpful (but not excessive) comments.
*   **SEO & Accessibility**: Every web page must have proper metadata, semantic HTML, and unique IDs.

## 4. Documentation Strategy

*   **README.md**: The project's homepage, always kept up-to-date.
*   **task.md**: The active TODO list for the current session.
*   **implementation_plan.md**: The design document for the current task.
